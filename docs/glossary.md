# Glossary

Terms used across Kleiber code and docs. When in doubt, link to this file rather than redefining a term.

---

**ADR (Architecture Decision Record).** A short markdown entry in [`architecture/decisions.md`](architecture/decisions.md) recording a significant design decision and its rationale. ADRs are append-only.

**Agent.** An AI software agent (Claude Code, Cursor, Aider, etc.) that performs tasks on this codebase. Agents follow [`agents/PROTOCOL.md`](agents/PROTOCOL.md).

**AI Bridge.** The Kleiber subsystem that talks to LLMs and the `gopls` MCP server, builds Go-aware prompts, and validates AI-proposed edits. Lives in `internal/ai/`.

**Buffer.** The in-memory representation of a file open in the editor. Owns text, undo history, and edit operations. Defined in `internal/editor/`.

**Command.** A discrete user-invokable action (e.g., "Save File", "Go to Definition"). Commands are dispatched by the Command Dispatcher and can be invoked from UI, keyboard, palette, or programmatically.

**Command Dispatcher.** The component that maps user input and palette selections to Command invocations. Lives in `internal/commands/`.

**Concurrency Visualizer.** A planned Kleiber feature (Milestone 4) that shows a live graph of goroutines and channels in a running Go process.

**Core.** The non-UI portion of Kleiber: editor engine, project model, LSP client, AI bridge, runtime monitor, command dispatcher, config. All written in Go and UI-agnostic.

**DAP (Debug Adapter Protocol).** Microsoft-standardized protocol for talking to debuggers. We use DAP to talk to Delve.

**Delve.** The de facto Go debugger. We use it via DAP.

**Editor Engine.** The subsystem responsible for buffers, views, cursors, undo history. Lives in `internal/editor/`.

**Event Bus.** The typed pub/sub system used for cross-component notifications inside the Core. Subscribers (especially the UI) react to events asynchronously.

**Fixture.** A small, often-broken Go project under `/fixtures/` used to drive integration tests. Fixtures are deliberately unusual; don't "fix" them.

**`gopls`.** The official Go language server, maintained by the Go team at Google. Kleiber uses `gopls` as its language brain via LSP and MCP.

**Inline annotation.** A visual hint rendered on top of source code: type hints, runtime metrics, errors, AI suggestions.

**LSP (Language Server Protocol).** Microsoft-standardized protocol for IDEs to talk to language servers. We use LSP to talk to `gopls`.

**LSP Client.** The Kleiber component that manages the `gopls` subprocess and speaks LSP. Lives in `internal/lsp/`.

**MCP (Model Context Protocol).** Anthropic-developed protocol for letting AI models interact with structured tools and data. `gopls` v0.20+ exposes an MCP server, which Kleiber consumes from the AI Bridge.

**Project.** A directory tree containing one or more Go modules, opened in Kleiber as a single workspace.

**Project Model.** The Kleiber subsystem representing the open project. Lives in `internal/project/`.

**Provider.** An interface implemented by AI vendors (Anthropic, OpenAI, Ollama). Defined in `internal/ai/`.

**Refactor Proposal.** A diff generated by the AI Bridge that must be compiler-validated before being shown to the user.

**Runtime Monitor.** The Kleiber subsystem connecting to Delve, parsing pprof, and (in future milestones) consuming OpenTelemetry and eBPF data. Lives in `internal/runtime/`.

**Subprocess.** Any process Kleiber spawns: `gopls`, Delve, `go build`, `go test`. Subprocesses are supervised and isolated from the main editor.

**Supervisor.** Code that watches a subprocess, restarts it on failure, and surfaces errors without crashing the editor.

**View.** The cursor and selection state of a Buffer in a particular window or pane. One Buffer can have multiple Views.
