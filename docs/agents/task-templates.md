# Task Templates

Templates for the four most common task shapes. Use these as the structure of your Step 1 task understanding (see [PROTOCOL §1](PROTOCOL.md)).

---

## Template A — New feature

```markdown
## Goal
<one sentence>

## User-visible change
<what does the user see or do differently after this lands?>

## Scope
- Packages touched:
- Files created:
- Files modified:

## Out of scope
-

## Acceptance criteria
- [ ] <observable check>
- [ ] <observable check>
- [ ] Tests added: <names>
- [ ] Docs updated: <paths>

## Open questions
-
```

---

## Template B — Bug fix

```markdown
## Bug
<what's wrong, with reproduction steps>

## Root cause
<one paragraph after investigation>

## Fix
<one paragraph: the approach>

## Regression test
<the test that fails before the fix and passes after>

## Acceptance criteria
- [ ] Regression test is added and passes
- [ ] All existing tests still pass
- [ ] No new lint warnings
```

---

## Template C — Refactor (no behavior change)

```markdown
## Goal
<why this refactor makes future work easier>

## Behavior change
None. (If there is one, this is not a refactor — use Template A.)

## Scope
- Packages touched:

## Verification that behavior didn't change
- [ ] Full test suite passes unchanged
- [ ] No new tests needed (or: list any that capture invariants we want to lock in)

## Acceptance criteria
- [ ] Tests pass with no test file modifications (or, if test files are renamed only, the diff shows that clearly)
- [ ] Public API surface unchanged (verify with a tool or manual check)
```

---

## Template D — Documentation change

```markdown
## What
<which docs are being updated>

## Why
<the trigger: new feature, outdated content, missing context>

## Scope
- Files touched:

## Acceptance criteria
- [ ] All internal links resolve
- [ ] Code examples compile (run `go build` on extracted snippets if any)
- [ ] No factual contradictions with other docs
```

## Report templates (for Step 5)

### Standard report

```markdown
## What changed
- `path/to/file.go`: <one line>
- `path/to/other.go`: <one line>

## Why (mapping to acceptance criteria)
- [x] AC1 — implemented in `file.go::FuncName`
- [x] AC2 — implemented in `other.go::Method`
- [x] AC3 — covered by `file_test.go::TestX`

## Tests run
- `go test ./internal/foo/...` — PASS (12 tests, 0.4s)
- `go test -race ./internal/foo/...` — PASS

## Verification checklist
- [x] go build ./... succeeds
- [x] go test ./... succeeds
- [x] gofmt -s -d . clean
- [x] go vet ./... clean
- [x] Docs updated: <paths>

## Open questions
- <thing the human should review>

## Generated by
<agent name + version>
```

### Blocked report

When you can't finish, use this:

```markdown
## Status
Blocked.

## What I did
<list what was attempted>

## Where it broke down
<one paragraph>

## What I need
<specific question or decision required from a human>

## Files in intermediate state
<paths that are partially modified — be honest>
```

A blocked report is much better than a fake-finished one.
