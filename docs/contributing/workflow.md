# Workflow

How code gets from idea to `main`.

## The short version

1. Open or pick up an issue.
2. Create a branch.
3. Write code with tests.
4. Open a PR using the template.
5. Get review (at least one approval).
6. CI must be green.
7. Squash merge.

## Branches

- `main` — always deployable. Protected. Direct pushes forbidden.
- `feature/<slug>` — human-authored feature branches.
- `agent/<task-id>-<slug>` — agent-authored branches.
- `fix/<slug>` — bug fixes.
- `docs/<slug>` — documentation-only changes.

## Commits

- One logical change per commit. Use `git add -p` if needed.
- Subject line: imperative present tense ("Add LSP client", not "Added LSP client" or "Adds LSP client").
- Subject line ≤ 72 characters.
- Body explains *why*, not *what*. The diff already shows *what*.
- Reference issues: `Refs #123` or `Fixes #123` in the body.

Example:

```
Add LSP client subprocess supervision

The previous LSP client assumed gopls would never crash. In practice
it does, especially on malformed go.mod files. This change adds a
restart-on-exit policy with exponential backoff, capped at 5 retries
in 60 seconds.

Refs #42
```

## Pull requests

### Opening a PR

Use the template from [`agents/PROTOCOL.md`](../agents/PROTOCOL.md) §4.3, which is:

```markdown
## What
## Why
## How
## Tests
## Docs
## Generated by (agent PRs only)
```

### Size

Aim for PRs under 400 lines of diff (excluding generated code and tests). If your change exceeds that, split it.

Acceptable reasons for a large PR: initial scaffolding of a new package, mechanical refactor with clear scope, generated code update.

### Review

- At least one human approval required.
- Reviewer checks: does it follow [`coding-standards.md`](coding-standards.md)? Are tests adequate? Are docs updated?
- Authors respond to all review comments — either by changing the code or by explaining why not.
- "LGTM" without comments is acceptable if the reviewer actually read the diff.

### CI

CI runs on every push to a PR:

- `go build ./...`
- `go test ./...`
- `go test -race ./...`
- `gofmt -s -d .` (must be empty)
- `go vet ./...`
- `staticcheck ./...`
- `golangci-lint run`

All must pass before merge.

### Merging

- Squash merge by default. The PR title becomes the commit message; ensure it's good.
- Merge commits are reserved for releases.
- Delete the branch after merge.

## Releases

We don't release yet (pre-alpha). When we do:

- Semantic versioning: `vMAJOR.MINOR.PATCH`.
- Pre-alpha: `v0.0.x`.
- Alpha: `v0.1.x` through `v0.9.x`.
- Beta: `v0.x.y` with explicit beta tag.
- First stable: `v1.0.0`.
- Changelog at `CHANGELOG.md` follows [Keep a Changelog](https://keepachangelog.com/).
- Tags are signed.

## Issues

### Labels

- `type/bug` — something broken.
- `type/feature` — new capability.
- `type/refactor` — improving structure, no behavior change.
- `type/docs` — documentation.
- `type/chore` — tooling, CI, build.
- `priority/{p0,p1,p2,p3}` — p0 is "drop everything", p3 is "someday".
- `area/{editor,lsp,ai,runtime,ui,project,...}` — which subsystem.
- `good first issue` — approachable for newcomers.
- `roadmap` — relates to the public roadmap.
- `agent-friendly` — well-scoped for an AI agent to attempt.

### Lifecycle

1. Open → triage (label, assign milestone).
2. Triage → ready (acceptance criteria written).
3. Ready → in progress (someone is working on it).
4. In progress → in review (PR open).
5. In review → done (merged).
6. Done → closed.

## Communication

- Async by default. Most decisions happen on PRs and issues.
- Long-form design happens in `docs/architecture/decisions.md` (ADRs).
- Quick questions: the project chat (link TBD).
- Security issues: see `SECURITY.md` (TBD) — do not file public issues for security bugs.

## Code of conduct

We follow the [Contributor Covenant](https://www.contributor-covenant.org/). TL;DR: be kind, give people the benefit of the doubt, criticize ideas not people.
